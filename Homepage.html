JAVA

Java est un langage de programmation de haut niveau, basé sur les classes, orienté objet, conçu pour avoir le moins de dépendances d'implémentation possible.

Microservices

Les microservices, également connus sous le nom d'architecture microservices, sont un style architectural qui structure une application comme une collection de services.

L'architecture microservices permet une livraison rapide, fréquente et fiable de grandes applications complexes. Elle permet également à une organisation d'évoluer dans sa pile technologique.

Hexagonal Architecture

L'idée principale de l'architecture hexagonale est d'avoir une séparation claire de la logique via le principe de séparation des interfaces. De plus, l'architecture hexagonale crée des applications faiblement couplées grâce au concept de ports et adaptateurs.

Les ports se réfèrent à l'interface et les adaptateurs se réfèrent à sa classe d'implémentation.

Types de Récupération (Fetch Types)

EAGER: Ce type de récupération signifie que les données associées sont chargées immédiatement avec l'entité parent. Cela peut entraîner des chargements plus lourds si de grandes collections sont récupérées de cette manière. Voici des exemples :

  • OneToOne: L'entité associée est récupérée lors de la requête initiale. Par exemple, si une Personne a une Adresse associée, l'Adresse sera chargée avec la Personne.
  • ManyToOne: De manière similaire à OneToOne, l'entité associée est chargée avec l'entité parent. Par exemple, si plusieurs Commandes sont associées à un Client, les données du Client seront chargées avec chaque Commande.

LAZY: Ce type de récupération retarde le chargement des données associées jusqu'à ce qu'elles soient réellement nécessaires. Cela peut améliorer les performances en chargeant uniquement ce qui est requis. Voici des exemples :

  • OneToMany: La collection associée n'est pas chargée immédiatement. Par exemple, un Département peut avoir plusieurs Employés, mais les Employés ne sont récupérés que lorsqu'ils sont accédés.
  • ManyToMany: De manière similaire à OneToMany, la collection des entités associées est chargée à la demande. Par exemple, si des Étudiants sont associés à plusieurs Cours, les Cours ne sont chargés que lorsqu'ils sont nécessaires.

Comprendre ces types de récupération est crucial pour gérer efficacement les performances et l'utilisation des ressources. Par exemple, utiliser la récupération EAGER pour de grandes collections peut entraîner des problèmes de performance, tandis que la récupération LAZY peut provoquer une LazyInitializationException si les données sont accédées en dehors du contexte de transaction.

Je peux également considérer la performance et l'utilisation de la mémoire. Une grande différence est que la stratégie de récupération EAGER permet d'utiliser l'objet de données récupéré sans session. Pourquoi ?

Toutes les données sont récupérées lorsque les données marquées comme EAGER sont dans l'objet lorsque la session est connectée. Cependant, en cas de stratégie de chargement paresseux (LAZY), l'objet marqué comme paresseux ne récupère pas les données si la session est déconnectée (après l'instruction session.close()).

Tout cela peut être réalisé par le biais des proxys Hibernate. La stratégie EAGER permet aux données de rester disponibles après la fermeture de la session.
