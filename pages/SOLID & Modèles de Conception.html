SOLID : Principes de Conception Orientée Objet

Principe de Responsabilité Unique

- La classe fournit une fonctionnalité ciblée ou répond à un souci spécifique concernant une fonctionnalité.
- Ce principe stipule qu'une classe ne doit avoir qu'une seule raison de changer.
- Le principe de responsabilité unique indique que si nous avons plusieurs responsabilités, chacune doit avoir sa propre classe afin que, en cas de changement, notre code puisse être modifié de manière organisée.

Principe Ouvert/Fermé

- "Ouvert" dans le Principe Ouvert/Fermé signifie que les classes sont ouvertes à l'héritage et que les méthodes peuvent être surchargées, tandis que "fermé" signifie que les classes de base existantes ne doivent pas être modifiées.

Principe de Substitution de Liskov

- Nous devons pouvoir substituer des objets de la classe de base par des objets de la classe enfant, et cette substitution ne doit pas altérer le comportement ou les caractéristiques du programme.
- Le principe de substitution de Liskov est violé lorsque la classe enfant modifie complètement le comportement ou le contrat de la méthode de la classe de base en la surchargeant.

Principe de Ségrégation des Interfaces

- Créez des interfaces spécifiques au client et bien définies. Les clients ne doivent pas être contraints d'implémenter des interfaces qu'ils n'utilisent pas.
- Il consiste à diviser les interfaces plus grandes avec des méthodes non liées en elles en interfaces plus petites et hautement cohésives.
- Ce principe plaide pour la création d'interfaces plus petites et plus ciblées plutôt que des interfaces larges et monolithiques.
Évitez la pollution des interfaces.

Principe d’Inversion des Dépendances

- Les modules de haut niveau ne doivent pas dépendre des modules de bas niveau. Tous deux doivent dépendre d'abstractions.
- Les abstractions ne doivent pas dépendre des détails. Les détails doivent dépendre des abstractions.
- Cela signifie que le code ne doit pas créer lui-même des objets de toutes ses dépendances. Les dépendances doivent être fournies au code depuis l'extérieur.
- Il s'agit de la stratégie de dépendance sur des interfaces et des méthodes et classes abstraites plutôt que sur des classes et méthodes concrètes.

Design Patterns

Creational Patterns

Singleton

Ensures a class has only one instance and provides a global point of access to it (e.g., logging system).

Factory Method

Defines an interface for creating an object but lets subclasses decide which class to instantiate (e.g., shape factory).

Abstract Factory

Provides an interface for creating families of related or dependent objects without specifying their concrete classes (e.g., GUI toolkit).

Builder

Separates the construction of a complex object from its representation, allowing the same construction process to create various representations (e.g., pizza customization).

Prototype

Creates new objects by copying existing objects (prototypes) (e.g., cell replication).

Structural Patterns

Adapter

Allows incompatible interfaces to work together by wrapping one object with a different interface (e.g., media player adapter).

Bridge

Decouples an abstraction from its implementation, so the two can vary independently (e.g., shape and color).

Composite

Composes objects into tree structures to represent part-whole hierarchies. Lets clients treat individual objects and compositions of objects uniformly (e.g., employee hierarchy).

Decorator

Dynamically adds behaviors to objects without affecting other objects of the same class (e.g., coffee with toppings).

Facade

Provides a simplified interface to a complex system of classes, libraries, or frameworks (e.g., computer startup).

Flyweight

Minimizes memory usage by sharing as much data as possible with similar objects.

Proxy

The Proxy pattern provides a surrogate or placeholder for another object to control access to it.

Behavioral Patterns

Observer

Defines a one-to-many dependency between objects, so that when one object changes state, all its dependents are notified and updated automatically (e.g., weather station).

State

Allows an object to alter its behavior when its internal state changes. The object will appear to change its class (e.g., traffic light).

Strategy

Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Lets the algorithm vary independently from clients that use it (e.g., payment strategy).

Template Method

Defines the skeleton of an algorithm in a method, deferring some steps to subclasses. Lets subclasses redefine certain steps of an algorithm without changing the algorithm's structure (e.g., algorithm variations).

Visitor

Represents an operation to be performed on the elements of an object structure. Lets you define a new operation without changing the classes of the elements on which it operates (e.g., shipping cost calculation).
